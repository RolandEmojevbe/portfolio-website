Project Overview

To round of the eight months of the UX design introduction and immersion courses at Career Foundry, I spent the next two months diving into the world of frontend development, learning how to code this very portfolio website.

As a novice, with absolutely not prior frontend development experience, it has been a very steep learning curve. I am oh so proud of making it out the other side.

This case study looks at my process, from writing my very first line of code to pushing my own polished HTML, CSS and JavaScript files to a web host - so the finished responsive portfolio website you see today can be accessed by anyone with an internet connection.

1

Context and Challenge

Being a naturally curious person, I jumped at the opportunity to level up my skills as a designer and learn to code a website. By understanding the building blocks that make a website, I will recommend and build ideas based on feasible solutions, taking into consideration potential boundaries, all the while having smoother and faster hand-offs with developers.

Duration: Two months (Jan-March 2021).

My Roles: UX Researcher, UX Designer, UI Designer, Frontend developer.

Tools: Figma, Atom, Github, W3C Validator.

The Goal: Create a responsive, user-friendly website, that showcases my work, skills, and personality, from scratch, completely coded by me.

The Challenge: Starting with absolutely no front-end development knowledge whatsoever and learning HTML, CSS, and JavaScript in just 2 months.

2

Process and Insights

With my UX hat on I started where all well trained designers would: Scoping out the market space and researching the various types of portfolio websites for inspiration, common themes, and areas I could try to improve on.

I then decided to carry the tone, personality and branding style from my CV and emulated this in a prototype I created using Figma. This would keep a consistent theme across my professional brand.

Next I deep dived into all things HTML, CSS and JavaScript putting the learnings into practise by writing the code for my website in Atom. Using pain point display sizes in the CSS styling to ensure a responsive display across different devices.

I then tested with four participants and used cross-browser testing. I’ve implemented the critical findings and quick wins and will continue to iterate on my website in true design thinking style.

Finally I used several code linters (including w3C validation) to ensure the quality of my HTML, CSS and JavaScript code.

3

Solution and Results

I utilised task analysis and created user flows in order to define the information architecture of the app.

Through task analysis, I identified the common steps that would be required to perform and complete specific tasks within the app. I then built user flows to visualise exactly the path my users would take.

PROTOTYPE

CREATING VISUAL BLUEPRINTS OF MY PRODUCT

With a firm understanding of the problem, my users, the steps required to perform specific tasks and the routes my users would take, I was now able to start rapid prototyping. A small sample of screens are shared below.

Low-high-fidelity prototypes

Before moving on to high-fidelity prototyping, I conducted usability testing to ensure the core elements and functionality met users’ needs. My testing process is further explained in the next section (Test).

Build your programme

Inline with competitor research and user interviews, I chose to implement a build your programme feature.

This would allow a tailored and personalised experience for my users, better meeting their language learning needs.

Lesson 1

I chose to delay the registration process until the user had completed their first lesson. This would allow speculative users to get a feel for the app before committing to signing up.

This helps to increase user conversion as the benefit of registration is clear and the process seems like a small final step to completing the lesson and achieving their language learning goal.

TEST

UNDERSTANDING HOW PEOPLE INTERACT WITH MY PRODUCT

As I was rapid prototyping, following low-fidelity prototyping, I quickly moved on to usability testing in order to achieve validation with users early on in the process

Three moderated in-person tests were held at participants’ homes. Participants aged 18-35.

Full time Student.

Tried to learn a Nigerian language, visits a new country every 2 years.

Management consultant.

Learned French in school, visits a new country once a year.

Charted Accountant

Currently learning Spanish, visits a new country multiple times a year.

Using Jakob Nielsen’s error severity rating scale, I ranked the severity of the observations from usability testing and developed key recommendations on how to proceed.

Usability testing provided me with valuable insights to areas that I had originally overlooked. These recommendations were quickly implemented allowing me to improve the usability of my designs before I began high fidelity prototyping.

Example 1

Problem: Users had difficulty understanding the build my program questions which were a key feature of my app.

Solution: I included new screens giving optional example information to users should they need it.

Example 2

Problem: Users could not tell how exactly their program had been built. In other words, they were interested in the results of the questionnaire.

Solution: I included an additional screen summarising the findings and how the programme had been adapted as a result.

Project Retrospective

Overall, I had a lot of fun creating HoliPrep and it’s been nice looking back at the journey - starting from a blank sheet to a clickable prototype.

What went well?

I was able to understand the importance of UX research and in ensuring all decisions were based on findings or had a strong, valid reason for not listening to the data

Skills: UX research, User testing, Iterative design.

Process: User-centred design, decisive design.

solution: Do this more.

What didn’t go well?

I was unable to design all the features I wanted to include in the app. However, this means I had to prioritise the most important features for the MVP.

Skills: Wireframing, Rapid prototyping, iterative design

Process: Prioritising designs

Solution: Prioritise MVP elements at the start of the project and build a timetable that centres around the MVP.

What could be improved?

A lot of time was spent deciding UI elements of the design and creating elements that perfectly emulated Holiprep.

Skills: UI design, Attention to detail

Process: Polishing designs

Solution: Keep up to date with UI best practices and trends to develop a living mood board of inspiration.

Key Takeaway

It’s impossible to create the perfect solution on the first attempt and the power of iterative prototyping and usability testing in the design thinking process should not be underestimated.

Getting down ideas in low-fidelity wireframes was a very useful and simple way to test the logic and flows of the design. Even just by creating the initial draft of the prototype on paper, I immediately started to notice elements I had missed or not given much thought.

Testing with the target audience took the designs a step further and it was interesting to see that some of my views and logic (that I thought were crystal clear) weren’t always the case for my users.
