How might we design a mobile app that empowers people to learn a new language?

Project Overview

Holiprep is a native mobile app that arms travellers with a basic survival toolkit for communicating whilst on holiday. Holiprep was created as part of the UX Design intensive course program at CareerFoundry.

Duration: One month.

My Roles: UX Researcher, UX Designer, UI Designer.

Tools: Marvel POP App, Sketch, Photoshop.

Methodology

In order to stay focused on the needs of my users when designing the solution, I utilised the Design Thinking Process:

EMPATHIZE

GETTING TO KNOW THE PROBLEM SPACE

Without pre-existing insights into the market, I conducted an analysis of three competitor apps.

The main goal was to understand what solutions were currently being offered by competitors, and their pros and cons, in order to design a solution that exceeded them.

❮ ❯

Early Insights

Segmenting users by objectives, ie whether they were concerned only with learning new words rather than phrases, would help with driving the right kind of learning system to the apps.

The apps that stood out were those that focused on competencies (eg ordering a coffee, introducing yourself etc) and sentence forming, rather than just words.

Implementing a ‘build your programme’ feature would be very useful in understanding the current needs of users and their objectives. This would allow me to create a more targeted app.

If I were to create a learning app I would:

  1. Incorporate the ‘build your programme’ feature in order to understand the users needs and incorporate onboarding as part of the learning process.
  2. Incorporate the use of pictures acting out the words/phrases being learned, to add another element of visual stimulation for the users and aid in achieving the apps end goal
  3. Include a visual timeline so user can see how they are progressing against their goal.

User Interviews

I also conducted user research to gather qualitative information directly from the potential users of the app.

❮ ❯

Key Takeaways

From the user interviews, it was evident that users need a simple app that was more personalised and geared towards their holiday needs.

Users also usually studied in 10-15 minute windows on language learning apps. This provided an upper limit for a typically lesson duration.

DEFINE

IDENTIFYING MY KEY USERS & THEIR GOALS

Armed with the research and data from the empathize stage, I was able to create a primary persona of the user that would have the most impact on the app.

This allowed me to better understand the user’s needs, behaviours, and goals.

I defined a list of user stories to allow me to focus on designing functions and solutions from Katherine’s perspective.

  • As an active girl, I want lessons to be broken down into 10-15-minute windows so that I can learn on the go.
  • As I commute on the tube in London, I want to be able to download lessons so that I can learn offline and make my commute time more productive.
  • As a globe trotter, I want to learn the basics of the language so that I can survive and feel confident asking for help along the way.
  • As a solo traveller, I want to be able to improve my pronunciation and practise speaking the new language so that I can communicate well with the locals.

I also defined a list of job stories to provide the context in which Katherine’s users stories might take place.

  • When I am travelling the world alone, I want to know the basics of the local language so that I can feel safe in my surroundings and ask for help should I need it.
  • When I am commuting to work, I want to be able to use an app to learn a new language so that I can make my commute more productive.
  • When I am learning a new language, I want to be visually stimulated so that I feel more engaged in the process rather than just reading plain text.
  • When I am learning a new language, I want to be able to track my progress towards a specified end date so that I can ensure I am ready ahead of my holiday.

From the deeper insights (developed building the primary persona and stories) I defined a problem statement to help focus the project on Katherine’s need and hypothesise a solution.

Problem statement

Katherine needs a way to learn the basics of a new language on the go because she has a very busy work and social schedule and enjoys travelling the world solo. She needs to feel visually stimulated and be able to improve her verbal communication of the language so that she is able to ask for help should she need it.

We will know this to be true when we see that Katherine is able to practise speaking the new language within the app and study in short interval windows.

Hypothesis

We believe that by creating a visually oriented interface and lessons which are broken down into 10-15-miunte windows, that can also be downloaded for offline use for Katherine, we will achieve a way for Katherine’s basic knowledge of new languages to improve whilst juggling a busy work and social calendar.

IDEATE

OUTLINING THE JOURNEYS USERS TAKE TO ACHIEVE THEIR GOALS

I utilised task analysis and created user flows in order to define the information architecture of the app.

Through task analysis, I identified the common steps that would be required to perform and complete specific tasks within the app. I then built user flows to visualise exactly the path my users would take.

PROTOTYPE

CREATING VISUAL BLUEPRINTS OF MY PRODUCT

With a firm understanding of the problem, my users, the steps required to perform specific tasks and the routes my users would take, I was now able to start rapid prototyping. A small sample of screens are shared below.

Low-high-fidelity prototypes

Before moving on to high-fidelity prototyping, I conducted usability testing to ensure the core elements and functionality met users’ needs. My testing process is further explained in the next section (Test).

Build your programme

Inline with competitor research and user interviews, I chose to implement a build your programme feature.

This would allow a tailored and personalised experience for my users, better meeting their language learning needs.

Lesson 1

I chose to delay the registration process until the user had completed their first lesson. This would allow speculative users to get a feel for the app before committing to signing up.

This helps to increase user conversion as the benefit of registration is clear and the process seems like a small final step to completing the lesson and achieving their language learning goal.

TEST

UNDERSTANDING HOW PEOPLE INTERACT WITH MY PRODUCT

As I was rapid prototyping, following low-fidelity prototyping, I quickly moved on to usability testing in order to achieve validation with users early on in the process

Three moderated in-person tests were held at participants’ homes. Participants aged 18-35.

Full time Student.

Tried to learn a Nigerian language, visits a new country every 2 years.

Management consultant.

Learned French in school, visits a new country once a year.

Charted Accountant

Currently learning Spanish, visits a new country multiple times a year.

Using Jakob Nielsen’s error severity rating scale, I ranked the severity of the observations from usability testing and developed key recommendations on how to proceed.

Usability testing provided me with valuable insights to areas that I had originally overlooked. These recommendations were quickly implemented allowing me to improve the usability of my designs before I began high fidelity prototyping.

Example 1

Problem: Users had difficulty understanding the build my program questions which were a key feature of my app.

Solution: I included new screens giving optional example information to users should they need it.

Example 2

Problem: Users could not tell how exactly their program had been built. In other words, they were interested in the results of the questionnaire.

Solution: I included an additional screen summarising the findings and how the programme had been adapted as a result.

Project Retrospective

Overall, I had a lot of fun creating HoliPrep and it’s been nice looking back at the journey - starting from a blank sheet to a clickable prototype.

What went well?

I was able to understand the importance of UX research and in ensuring all decisions were based on findings or had a strong, valid reason for not listening to the data

Skills: UX research, User testing, Iterative design.

Process: User-centred design, decisive design.

solution: Do this more.

What didn’t go well?

I was unable to design all the features I wanted to include in the app. However, this means I had to prioritise the most important features for the MVP.

Skills: Wireframing, Rapid prototyping, iterative design

Process: Prioritising designs

Solution: Prioritise MVP elements at the start of the project and build a timetable that centres around the MVP.

What could be improved?

A lot of time was spent deciding UI elements of the design and creating elements that perfectly emulated Holiprep.

Skills: UI design, Attention to detail

Process: Polishing designs

Solution: Keep up to date with UI best practices and trends to develop a living mood board of inspiration.

Key Takeaway

It’s impossible to create the perfect solution on the first attempt and the power of iterative prototyping and usability testing in the design thinking process should not be underestimated.

Getting down ideas in low-fidelity wireframes was a very useful and simple way to test the logic and flows of the design. Even just by creating the initial draft of the prototype on paper, I immediately started to notice elements I had missed or not given much thought.

Testing with the target audience took the designs a step further and it was interesting to see that some of my views and logic (that I thought were crystal clear) weren’t always the case for my users.
